# <a name="project-migration-and-upgrade-reference-for-visual-studio"></a>Informazioni di riferimento per la migrazione e l'aggiornamento di un progetto per Visual Studio

::: moniker range="vs-2017"

Ogni versione di Visual Studio in genere supporta la maggior parte dei tipi precedenti di progetti, file e altri asset. È possibile lavorare [](../ide/solutions-and-projects-in-visual-studio.md)con questi elementi come sempre e, a condizione che non si dipendono da funzionalità più recenti, Visual Studio tenta di mantenere la compatibilità con le versioni precedenti, ad esempio Visual Studio 2015, Visual Studio 2013 e Visual Studio 2012. Vedere le [note sulla versione](/visualstudio/releasenotes/vs2017-relnotes/) per le funzionalità specifiche di ogni versione.

Varia periodicamente anche il supporto per alcuni tipi di progetto. È possibile che una versione più recente di Visual Studio non supporti più alcuni progetti o richieda l'aggiornamento di un progetto e di conseguenza non sia più compatibile con le versioni precedenti. Per visualizzare lo stato corrente sui problemi di migrazione, consultare il [sito Visual Studio Developer Community](https://aka.ms/feedback/suggest?space=8).

Questo articolo offre informazioni dettagliate relative solo ai tipi di progetto di cui Visual Studio 2017 può eseguire la migrazione. L'articolo esclude i tipi di progetto che non sono più supportati in Visual Studio 2017 e di cui non è possibile eseguire la migrazione. L'articolo esclude anche i tipi di progetto supportati che non evidenziano problemi di migrazione. L'elenco è disponibile in [Selezione della piattaforma e compatibilità](/visualstudio/productinfo/vs2017-compatibility-vs).

> [!IMPORTANT]
> Determinati tipi di progetto richiedono l'installazione dei carichi di lavoro appropriati usando il programma di installazione di Visual Studio. Se non è installato il carico di lavoro, Visual Studio non segnala un tipo di progetto sconosciuto o incompatibile. In tal caso, controllare le opzioni di installazione e riprovare. Anche in questo caso, vedere l'articolo [Selezione della piattaforma e compatibilità](/visualstudio/productinfo/vs2017-compatibility-vs) per informazioni dettagliate sul supporto del progetto in Visual Studio 2017.

## <a name="how-visual-studio-decides-when-to-migrate-a-project"></a>In che modo Visual Studio decide di eseguire la migrazione di un progetto

In ogni nuova versione di Visual Studio, in genere, si cerca di mantenere la compatibilità con le versioni precedenti, in modo che lo stesso progetto possa essere aperto, modificato e compilato in diverse versioni. Tuttavia, nel corso del tempo, esistono modifiche inevitabili che rendono alcuni tipi di progetto non più supportati. Vedere [Platform Targeting and Compatibility for](/visualstudio/productinfo/vs2017-compatibility-vs) which project types are supported in Visual Studio 2017 (Scelta della piattaforma e compatibilità per i tipi di progetto supportati in Visual Studio 2017). In questi casi, una versione più recente Visual Studio non carica il progetto e non offre un percorso di migrazione. È necessario mantenere il progetto in una versione precedente di Visual Studio che lo supporta.

In altri casi, si riuscirà ad aprire il progetto con la versione più recente di Visual Studio, ma sarà necessario aggiornare o migrare il progetto in un modo che potrebbe renderlo incompatibile con le versioni precedenti. In Visual Studio viene usata una serie di criteri per determinare se tale migrazione è necessaria:

- Compatibilità con le versioni di destinazione delle piattaforme, fino a Visual Studio 2013 RTM.

- Compatibilità degli asset della fase di progettazione con le versioni precedenti di Visual Studio. (ovvero canali diversi di Visual Studio 2017; Visual Studio 2015 RTM & Update 3; Visual Studio 2013 RTM & Update 5; Visual Studio 2012 Update 4; Visual Studio 2010 SP 1. Visual Studio 2017 mira a non riuscire normalmente con asset deprecati della fase di progettazione senza danneggiarli, in modo che le versioni precedenti possano comunque aprire il progetto.

- Se gli asset in fase di progettazione interrompono la compatibilità con versioni precedenti fino a Visual Studio 2013 RTM & Update 5.

Il proprietario dell'engineering del tipo di progetto in questione esamina questi criteri e prende la decisione in merito a supporto, compatibilità e migrazione. Ancora una volta, se possibile, si tenta di mantenere una compatibilità trasparente tra le versioni di Visual Studio, vale a dire consentire di creare e modificare progetti in una versione di Visual Studio che rimangano funzionanti anche in altre versioni.

Tuttavia, se tale compatibilità non può essere ottenuta, come accade con alcuni tipi di progetto descritti in questo articolo, Visual Studio apre la procedura guidata Aggiorna per apportare le necessarie modifiche unidirezionali.

Tali modifiche unidirezionali possono comportare la modifica della proprietà `ToolsVersion` nel file di progetto, che indica esattamente la versione di MSBuild che può trasformare il codice sorgente del progetto negli elementi eseguibili e distribuibili desiderati. Vale a dire, ciò che rende un progetto incompatibile con le versioni precedenti di Visual Studio non è la versione di *Visual Studio* in sé, bensì la versione di *MSBuild* come determinato dal valore `ToolsVersion`. Fintanto che la propria versione di Visual Studio contiene la toolchain di MSBuild che corrisponde al valore `ToolsVersion` di un progetto, Visual Studio può richiamare tale toolchain per compilare il progetto.

Per mantenere la massima compatibilità con i progetti creati in versioni precedenti, Visual Studio 2017 include le toolchain di MSBuild necessarie per supportare `ToolsVersion` 15, 14, 12 e 4. I progetti che usano uno di questi valori `ToolsVersion` generalmente vengono compilati correttamente. Ancora una volta, l'esito dipende dal supporto o meno del tipo di progetto da parte di Visual Studio 2017, come descritto in [Selezione della piattaforma e compatibilità](/visualstudio/productinfo/vs2017-compatibility-vs).

In questo contesto, si presenta naturalmente la questione se l'utente debba aggiornare manualmente il progetto o eseguire la migrazione a un valore `ToolsVersion` più recente. Tale cambiamento non è necessario e rischia di generare molti errori e avvisi che sarà necessario correggere per poter compilare di nuovo il progetto. Inoltre, se in Visual Studio il supporto per un valore `ToolsVersion` specifico viene eliminato in futuro, l'apertura del progetto attiverà il processo di migrazione del progetto precisamente perché il valore `ToolsVersion` deve essere modificato. In tal caso, il sottosistema per quel tipo di progetto specifico sa esattamente ciò che deve essere modificato e può apportare tali modifiche automaticamente come descritto in precedenza in questo articolo.

## <a name="migrate-a-project"></a>Eseguire la migrazione di un progetto

Mentre si tenta di mantenere la compatibilità con le versioni precedenti, possono essere presenti modifiche che non sono compatibili con le versioni precedenti. Vedere [Targeting piattaforma e compatibilità](/visualstudio/releases/2019/compatibility) per i tipi di progetto supportati in Visual Studio 2019. In questo caso, una versione più recente Visual Studio non carica il progetto o non offre un percorso di migrazione. Potrebbe essere necessario mantenere il progetto in una versione precedente di Visual Studio.

In alcuni casi, la versione più recente di Visual Studio può aprire un progetto, ma deve aggiornare o eseguire la migrazione del progetto in modo da renderlo incompatibile con le versioni precedenti. In Visual Studio viene usata una serie di criteri per determinare se tale migrazione è necessaria:

- Compatibilità con le versioni di destinazione delle piattaforme, fino a Visual Studio 2013 RTM.

- Compatibilità degli asset della fase di progettazione con le versioni precedenti di Visual Studio. (Ovvero canali diversi di Visual Studio 2019, Visual Studio 2017; Visual Studio 2015 RTM & Update 3; Visual Studio 2013 RtM & Update 5; Visual Studio 2012 Update 4; Visual Studio 2010 SP 1. Visual Studio 2019 mira a non riuscire correttamente con asset deprecati in fase di progettazione senza danneggiarli, in modo che le versioni precedenti possano comunque aprire il progetto.

- Se gli asset in fase di progettazione interrompono la compatibilità con versioni precedenti fino a Visual Studio 2013 RTM & Update 5.

Il team di progettazione proprietario del tipo di progetto esamina questi criteri ed effettua la chiamata in base al supporto, alla compatibilità e alla migrazione. Anche in questo caso, si tenta di mantenere la compatibilità tra le versioni Visual Studio in modo che, quando si creano e si modificano progetti in una versione di Visual Studio, funzioni solo in altre versioni.

In alcuni casi, la compatibilità non è possibile. Quindi, Visual Studio'aggiornamento guidato per apportare le modifiche unidireive necessarie. Queste modifiche unidireli possono comportare la modifica della proprietà nel file di progetto, che indica esattamente quale versione di MSBuild può trasformare il codice sorgente del progetto negli elementi eseguibili e distribuibili `ToolsVersion` desiderati.

Ciò che rende un progetto incompatibile con le  versioni precedenti Visual Studio non  è la versione Visual Studio, ma la MSBuild, come determinato da `ToolsVersion` . Se la versione di Visual Studio contiene la toolchain MSBuild corrispondente a in un progetto, Visual Studio possibile richiamare tale toolchain per compilare `ToolsVersion` il progetto.

Per mantenere la compatibilità con i progetti creati nelle versioni precedenti, Visual Studio 2019 include le toolchain MSBuild necessarie per supportare `ToolsVersion` 15, 14, 12 e 4. I progetti che usano uno di questi valori `ToolsVersion` generalmente vengono compilati correttamente. (Oggetto, anche in questo caso, se Visual Studio 2019 supporta il tipo di progetto, come descritto in [Platform Targeting and Compatibility](/visualstudio/releases/2019/compatibility).)

Si potrebbe essere tentati di aggiornare o eseguire manualmente la migrazione di un progetto a un valore `ToolsVersion` più recente. Non è necessario apportare una modifica di questo tipo e probabilmente genererebbe molti errori e avvisi che è necessario correggere per compilare nuovamente il progetto. Inoltre, se Visual Studio non supporta una specifica in futuro, il progetto attiva il processo di migrazione del progetto quando lo si apre perché il relativo valore `ToolsVersion` `ToolsVersion` deve essere modificato.

## <a name="migrate-a-project"></a>Eseguire la migrazione di un progetto

Anche se si tenta di mantenere la compatibilità con le versioni precedenti, possono essere presenti modifiche che non sono compatibili con le versioni precedenti. <!--(See [Platform Targeting and Compatibility](/visualstudio/releases/2019/compatibility) for which project types are supported in Visual Studio 2022.)--> In questo caso, una versione più recente di Visual Studio non carica il progetto né offre un percorso di migrazione. Potrebbe essere necessario gestire il progetto in una versione precedente di Visual Studio.

In alcuni casi, la versione più recente di Visual Studio può aprire un progetto, ma deve aggiornare o eseguire la migrazione del progetto in un modo che potrebbe renderlo incompatibile con le versioni precedenti. In Visual Studio viene usata una serie di criteri per determinare se tale migrazione è necessaria:

- Compatibilità con le versioni di destinazione delle piattaforme, fino a Visual Studio 2013 RTM.

- Compatibilità degli asset della fase di progettazione con le versioni precedenti di Visual Studio. Ovvero canali diversi di Visual Studio 2022, Visual Studio 2019, Visual Studio 2017, Visual Studio 2015 RTM & Update 3, Visual Studio 2013 RTM & Update 5, Visual Studio 2012 Update 4 e Visual Studio 2010 SP1. Visual Studio 2022 ha lo scopo di non riuscire normalmente con asset deprecati della fase di progettazione senza danneggiarli, in modo che le versioni precedenti possano comunque aprire il progetto.

- Se gli asset in fase di progettazione interrompono la compatibilità con versioni precedenti fino a Visual Studio 2013 RTM & Update 5.

Il team di progettazione proprietario del tipo di progetto esamina questi criteri ed effettua la chiamata per quanto riguarda il supporto, la compatibilità e la migrazione. Anche in questo caso, si tenta di mantenere la compatibilità tra le versioni di Visual Studio in modo che, quando si creano e si modificano progetti in una versione di Visual Studio, funzioni solo in altre versioni.

In alcuni casi, la compatibilità non è possibile. Quindi, Visual Studio l'aggiornamento guidato per apportare le modifiche unidire utente necessarie. Queste modifiche unidireli possono comportare la modifica della proprietà nel file di progetto, che indica esattamente quale versione di MSBuild può trasformare il codice sorgente del progetto negli elementi eseguibili e distribuibili `ToolsVersion` desiderati.

Ciò che rende un progetto incompatibile con le versioni precedenti di Visual Studio  non è la *versione Visual Studio,* ma la MSBuild precedente, come determinato da `ToolsVersion` . Se la versione di Visual Studio contiene la toolchain MSBuild corrispondente a in un progetto, Visual Studio può richiamare tale toolchain per compilare `ToolsVersion` il progetto.

Per mantenere la compatibilità con i progetti creati nelle versioni precedenti, Visual Studio 2019 include le toolchain MSBuild necessarie per supportare `ToolsVersion` le versioni 15, 14, 12 e 4. I progetti che usano uno di questi valori `ToolsVersion` generalmente vengono compilati correttamente. (Oggetto, anche in questo caso, se Visual Studio 2019 supporta il tipo di progetto, come descritto in Platform [Targeting and Compatibility](/visualstudio/releases/2019/compatibility).)

È possibile che si sia tentati di aggiornare manualmente un progetto o di eseguirne la migrazione a un valore `ToolsVersion` più recente. Non è necessario apportare una modifica di questo tipo e probabilmente genererebbe molti errori e avvisi che è necessario correggere per far ricompilare il progetto. Inoltre, se Visual Studio non supporta uno specifico in futuro, il progetto attiva il processo di migrazione del progetto quando lo si apre perché il relativo valore `ToolsVersion` `ToolsVersion` deve essere modificato.
